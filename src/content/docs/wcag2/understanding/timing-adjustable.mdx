### 의도
이 성공 기준의 목적은 장애가 있는 사용자가 가능할 때마다 웹 콘텐츠와 상호 작용할 수 있는 적절한 시간을 갖도록 보장하는 것이다. 시각 장애, 저시력, 기민성 장애(<span lang="en">dexterity impairments</span>), 인지 장애 등이 있는 사람은 콘텐츠를 읽거나 온라인 양식 작성과 같은 기능을 수행하는 데 더 많은 시간이 필요할 수 있다. 만약 웹 기능이 시간에 종속적이라면, 일부 사용자는 시간 제한이 발생하기 전에 필요한 작업을 수행하기 어려울 것이다. 이로 인해 서비스에 접근하지 못할 수도 있다. 시간에 종속되지 않는 기능을 설계하면 장애가 있는 사람이 이러한 기능을 성공적으로 완료하는 데 도움이 된다. 시간 제한을 비활성화하거나, 시간 제한 길이를 사용자 정의하거나, 시간 제한이 발생하기 전에 추가 시간을 요청하는 옵션을 제공하면, 예상보다 더 많은 시간이 필요한 사용자가 작업을 성공적으로 완료하는 데 도움이 된다. 이러한 옵션은 사용자에게 가장 유용한 순서대로 나열된다. 시간 제한을 비활성화하는 것이 시간 제한 길이를 사용자 정의하는 것보다 낫고, 이는 시간 제한이 발생하기 전에 추가 시간을 요청하는 것보다 낫다. 

사용자가 시작하지 않았으나 설정된 시간 이후 또는 주기적으로 발생하는 모든 과정은 시간 제한이다. 여기에는 콘텐츠의 부분 또는 전체 업데이트(예: 페이지 새로 고침), 콘텐츠 변경, 또는 사용자가 입력 요청에 반응할 수 있는 기회의 만료 등이 포함된다.

또한 사용자가 읽거나 이해할 수 있는 속도를 넘어서는 속도로 진행되거나 업데이트되는 콘텐츠도 포함된다. 즉, 애니메이션, 움직이거나 스크롤되는 콘텐츠는 사용자가 콘텐츠를 읽을 수 있는 시간을 제한한다.

이 성공 기준은 일반적으로 정보와 데이터가 조정 가능하거나 최종 사용자가 제어할 수 있는 한, 콘텐츠가 반복되거나 다른 콘텐츠와 동기화되는 경우에는 적용되지 않는다. 이 성공 기준이 적용되지 않는 시간 제한의 예로는 반복되는 텍스트 스크롤, 자막, 캐러셀이 있다. 이러한 상황에는 시간 제한이 포함되지만, 2.2.2 일시정지, 중지, 숨기기에 명시된 대로 콘텐츠에 접근할 수 있는 제어 기능이 있으므로 사용자는 여전히 콘텐츠를 이용할 수 있다.

그러나 경우에 따라 (예: 경매 또는 기타 실시간 이벤트) 시간 제한을 변경하는 것이 불가능할 수 있으며, 이러한 경우가 예외이다.

타이머에 의존하지 않는 대안이 있는 경우, 타이머로 작동하는 콘텐츠는 시간 조정을 할 필요가 없다. 예를 들어, 이메일 클라이언트와 같은 웹 애플리케이션은 인터페이스 오른쪽 하단에 임시 메시지('토스트' 메시지 등)로 새 이메일 도착 알림을 제공하고, 이 메시지는 5초 후에 사라진다. 사용자는 '받은 편지함' 보기와 같은 다른 방법을 통해 이메일 도착을 식별할 수 있으므로, 메시지가 사라지는 것이 새 메일 도착 여부를 판단하는 능력에 시간 제한을 설정하지 않는다. 만약 사용자가 동일한 정보를 발견하거나 동일한 기능을 수행할 다른 수단이 없다면, 각 메시지는 사용자가 정보에 접근할 수 있는 충분한 시간을 제공하기 위해 이 성공 기준을 충족해야 한다.

#### 시간 제한에 관한 참고 사항

*   시간이 정해진 서버 리디렉션은 아래 '일반적인 오류(<span lang="en">Common Failures</span>)'에서 찾을 수 있다.
*   시간이 정해지지 않은 서버 리디렉션(예: 3xx 응답 코드)은 시간 제한이 없으므로(즉시 작동하므로) 적용되지 않는다.
*   이 성공 기준은 콘텐츠 자체에 의해 설정된 시간 제한에만 적용된다. 예를 들어, 보안 문제를 해결하기 위해 시간 제한이 포함된 경우, 이는 해당 콘텐츠에 대한 표현 및 상호 작용 경험의 일부로 설계되었으므로 콘텐츠에 의해 설정된 것으로 간주한다. 사용자 에이전트나 인터넷 고유의 요인과 같이 콘텐츠 외부에서 설정된 시간 제한은 저작자의 통제하에 있지 않으며 WCAG 준수(Conformance) 요구 사항의 대상이 아니다. 웹 서버에 의해 설정된 시간 제한은 저작자/조직의 통제하에 있어야 하며 여기에 포함된다. [성공 기준 2.2.3 시간 제한 없음](no-timing), [2.2.4 방해 금지](interruptions), [2.2.5 재인증](re-authenticating)이 적용될 수도 있다.
*   시간 제한이 있는 2단계 인증 토큰과 같이 보안상의 이유로 구현된 특정 시간 제한은 필수적인 것으로 간주되어 이 기준에서 면제될 수 있다. 그러나 [3.3.7 중복 입력](redundant-entry), [3.3.8 접근 가능한 인증(최소)](accessible-authentication-minimum), [3.3.9 접근 가능한 인증(향상된)](accessible-authentication-enhanced)과 같은 다른 기준이 적용될 수 있다.
*   기본값의 10배는 임상 경험과 다른 지침 을 바탕으로 선택되었다. 예를 들어, 사용자가 응답하고 스위치를 누르는 데 15초가 허용된다면, 150초는 거의 모든 사용자가  (문제가 있더라도) 스위치를 누르기에 충분한 시간이다. 
*   20초 또한 임상 경험과 다른 지침을 바탕으로 한 것이다. '아무 키나 누르는 데' 20초는 운동 장애가 있는 사용자를 포함하여 거의 모든 사용자에게 충분하다. 일부 사용자는 실패할 수 있고 일부 사용자는 많은 시간이 주어지더라도 실패할 수 있다. 일부 애플리케이션의 경우, 임의로 제공하는 긴 시간은 장애인을 포함한 모든 사용자에게 보안 위험을 초래할 수 있으므로, 추가 시간을 요청할 적절한 시간이 필요하다. 예를 들어, 금융 거래에 사용되는 키오스크나 단말기의 경우, 사람들이 로그아웃하지 않고 자리를 떠나는 경우가 매우 흔하다. 이는 뒤에 오는 사람들에게 취약점을 노출한다. 비활성 기간이 길어진 후에야 묻고, 그 후 사람이 현재 있는지 표시하는 데 긴 시간을 제공하면 단말기가 오용될 수 있다. 활동이 없으면 시스템은 사용자가 거기에 있는지 물어야 한다. 그런 다음 사람이 거기에 있다는 표시('아무 키나 누르시오')를 요청하고, 거의 모든 사람이 응답할 수 있을 만큼 충분히 기다려야 한다. "아무 키나 누르시오"의 경우 20초면 이를 충족한다. 만약 사람이 여전히 존재한다고 표시하면, 장치는 질문하기 전의 정확한 상태로 돌아가야 한다.
*   20시간은 깨어 있는 하루보다 길기 때문에 상한선으로 선택했다.

시간 제한이 본질적인 요구 사항은 아니지만 사용자에게 시간 제한 이벤트(Event) 에 대한 제어권을 부여하면 결과가 무효화되는 경우, 제3자가 사용자에 대한 제한 시간을 조정할 수 있다(예: 시험 에서 두 배의 시간 부여).

[2.2.3: 시간 제한 없음](no-timing) 또한 참고하라.

### 이점

* 지체 장애가 있는 사람들은 반응하고, 입력하고, 활동을 완료하는 데 더 많은 시간이 필요한 경우가 많다. 저시력자는 화면에서 사물을 찾고 읽는 데 더 많은 시간이 필요하다. 시각 장애가 있고 스크린 리더를 사용하는 사람들은 화면 레이아웃을 이해하고, 정보를 찾고, 컨트롤을 작동하는 데 더 많은 시간이 필요할 수 있다. 인지적 또는 언어적 제약이 있는 사람은 읽고 이해하는 데 더 많은 시간이 필요하다. 청각 장애가 있고 수어로 의사소통하는 사람은 (일부에게는 제2언어일 수 있는) 텍스트로 인쇄된 정보를 읽는 데 더 많은 시간이 필요할 수 있다. 
* 수어 통역사가 청각 장애가 있는 사용자에게 오디오 콘텐츠를 전달하는 상황에서도 제한 시간을 조정하는 ​​것도 중요하다.
* 정보를 읽거나 이해하는 데 더 많은 시간이 필요할 수 있는 읽기 장애, 인지 장애, 학습 장애가 있는 사람들은 콘텐츠를 일시 정지하여 정보를 읽을 수 있는 추가 시간을 가질 수 있다

### 예제

* 웹사이트가 클라이언트 측 시간 제한을 사용하여 컴퓨터에서 자리를 비울 수 있는 사용자를 보호한다. 일정 기간 비활성상태가 지속된 후, 웹 페이지는 사용자에게 시간이 더 필요한지 묻는다. 응답이 없으면 시간이 만료된다.
* 웹 페이지에 최신 헤드라인이 자동 갱신되어 순환하는 필드가 있다. 사용자가 각 갱신 사이의 시간을 기본값의 10배까지 연장할 수 있도록 하는 상호작용 컨트롤이 있다. 이 컨트롤은 마우스나 키보드로 조작할 수 있다.
* 웹 페이지에 텍스트가 나타났다가 사라지는 애니메이션이 포함되어 있다. 어떤 경우에는 텍스트가 화면을 가로질러 스크롤되고, 다른 경우에는 배경으로 사라지기 전에 잠깐 동안만 표시된다. 페이지에는 텍스트가 사라지기 전에 읽기 어려운 사용자가 읽을 수 있도록 일시 중지 버튼이 포함되어 있다.
* 경매에서는 사용자가 입찰을 제출해야 하는 제한 시간이 있다. 제한 시간은 특정 품목에 입찰하려는 모든 사용자에게 적용되므로, 특정 사용자 한 명을 위해 제한 시간을 연장하는 것은 불공평할 것이다. 따라서 이러한 유형의 활동에는 제한 시간이 필요하며, 이 성공 기준에 따라 제한 시간의 연장, 조정 또는 비활성화가 요구되지 않는다. 
* 온라인 티켓 구매 사이트는 사용자가 좌석을 확정할 수 있도록 2분을 제공한다. 이러한 사이트의 티켓은 빨리 매진될 수 있으므로, 그보다 더 오래 티켓을 보유하는 것은 사이트의 본질을 무효화할 수 있다. 따라서 이는 제한 시간이 필수적이며 활동을 무효화하지 않고는 연장할 수 없는 경우이다. 
* 그러나 이 사이트는 가능한 많은 과정을 제한 시간 밖으로 배치한다. 예를 들어, 중요한 단계에 들어가기 전에 사용자가 이름, 결제 방법 등과 같은 필요한 정보를 제공할 수 있도록 허용한다. 
* 티켓 구매 사이트는 사용자가 선택한 좌석의 구매를 확인하는 데 2분을 허용하지만, 시간이 거의 다 되었을 때 사용자에게 경고하고 "시간 연장" 버튼을 클릭하는 것과 같은 간단한 동작으로 이 시간 제한을 여러 번 연장할 수 있도록 허용한다.

### 기법

이 섹션의 각 번호가 매겨진 항목은 접근성 지침 실무 그룹이 이 성공 기준을 충족하기에 충분하다고 판단하는 기법 또는 기법의 조합을 나타낸다. 기법은 기준의 최소 요구 사항을 넘어설 수 있다. 이러한 기법에서 포괄하지 않은 기준 충족의 다른 방법이 있을 수 있다. 다른 기법 사용에 대한 정보는 [WCAG 성공 기준에 대한 기법 이해](../understanding-techniques), 특히 "기타 기법" 섹션을 참조하라.

#### 충분 기법

아래에서 귀하의 콘텐츠와 일치하는 상황을 선택하라. 각 상황에는 해당 상황에 충분하다고 알려져 있고 문서화된 기법 또는 기법의 조합이 포함된다.

##### 상황 A: If there are session time limits:

*   [G133: Providing a checkbox on the first page of a multipart form that allows users to ask for longer session time limit or no session time limit](https://www.w3.org/WAI/WCAG22/Techniques/general/G133)
*   [G198: Providing a way for the user to turn the time limit off](https://www.w3.org/WAI/WCAG22/Techniques/general/G198)

##### 상황 B: If a time limit is controlled by a script on the page:

*   [G198: Providing a way for the user to turn the time limit off](https://www.w3.org/WAI/WCAG22/Techniques/general/G198)
*   [G180: Providing the user with a means to set the time limit to 10 times the default time limit](https://www.w3.org/WAI/WCAG22/Techniques/general/G180)
*   [SCR16: Providing a script that warns the user a time limit is about to expire](https://www.w3.org/WAI/WCAG22/Techniques/client-side-script/SCR16) **AND** [SCR1: Allowing the user to extend the default time limit](https://www.w3.org/WAI/WCAG22/Techniques/client-side-script/SCR1)

##### 상황 C: If there are time limits on reading:

*   [G4: Allowing the content to be paused and restarted from where it was paused](https://www.w3.org/WAI/WCAG22/Techniques/general/G4)
*   [G198: Providing a way for the user to turn the time limit off](https://www.w3.org/WAI/WCAG22/Techniques/general/G198)
*   [SCR33: Using script to scroll content, and providing a mechanism to pause it](https://www.w3.org/WAI/WCAG22/Techniques/client-side-script/SCR33)
*   [SCR36: Providing a mechanism to allow users to display moving, scrolling, or auto-updating text in a static window or area](https://www.w3.org/WAI/WCAG22/Techniques/client-side-script/SCR36)

#### 오류

다음은 접근성 지침 실무 그룹에서 이 성공 기준의 실패로 간주하는 일반적인 실수이다.

*   [F40: Failure due to using meta redirect with a time limit](https://www.w3.org/WAI/WCAG22/Techniques/failures/F40)
*   [F41: Failure of Success Criterion 2.2.1, 2.2.4, and 3.2.5 due to using meta refresh to reload the page](https://www.w3.org/WAI/WCAG22/Techniques/failures/F41)
*   [F58: Failure of Success Criterion 2.2.1 due to using server-side techniques to automatically redirect pages after a time-out](https://www.w3.org/WAI/WCAG22/Techniques/failures/F58)
