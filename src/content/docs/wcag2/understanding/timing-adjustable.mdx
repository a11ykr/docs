<dl>

<dd>사용자가 시간 제한을 끄거나 조정하거나 연장할 수 있다.</dd>

### 의도

이 성공 기준의 목적은 장애가 있는 사용자에게 가능할 때마다 웹 콘텐츠와 상호 작용할 수 있는 적절한 시간을 제공하는 것이다. 시각 장애, 저시력, 조작 능력 장애(<span lang="en">dexterity impairments</span>), 인지 장애 등의 장애가 있는 사람은 콘텐츠를 읽거나 온라인 양식 작성과 같은 기능을 수행하는 데 더 많은 시간이 필요할 수 있다. 웹 기능이 시간 의존적이라면 일부 사용자는 시간 제한이 발생하기 전에 필요한 작업을 수행하기 어려울 것이다. 이로 인해 서비스에 액세스할 수 없게 될 수 있다. 시간에 종속되지 않는 기능을 설계하면 장애인이 이러한 기능을 성공적으로 완료하는 데 도움이 된다. 시간 제한을 비활성화하거나, 시간 제한을 사용자 정의하거나, 시간 제한이 발생하기 전에 추가 시간을 요청하는 옵션을 제공하면 예상보다 더 많은 시간이 필요한 사용자가 작업을 성공적으로 완료하는 데 도움이 된다. 이러한 옵션은 사용자에게 가장 유용한 순서대로 나열된다. 시간 제한을 비활성화하는 것이 시간 제한 길이를 사용자 정의하는 것보다 낫다. 이는 시간 제한이 발생하기 전에 추가 시간을 요청하는 것보다 낫다.

사용자가 시작하지 않았으나 설정된 시간 이후 또는 주기적으로 발생하는 모든 과정은 시간 제한이다. 여기에는 콘텐츠의 부분 또는 전체 업데이트(예: 페이지 새로 고침), 콘텐츠 변경, 사용자가 입력 요청에 반응할 수 있는 기회의 시간(window of opportunity) 만료 등이 포함된다.

또한 사용자가 읽거나 이해할 수 있는 속도를 넘어서는 속도로 진행하거나 업데이트되는 콘텐츠도 포함된다. 즉, 애니메이션, 이동 또는 스크롤 콘텐츠는 사용자가 콘텐츠를 읽을 수 있는 능력에 시간 제한을 부여한다.

이 성공 기준은 일반적으로 정보와 데이터를 조정 가능하거나 최종 사용자가 제어할 수 있는 한 콘텐츠가 반복되거나 다른 콘텐츠와 동기화되는 경우 적용되지 않는다. 이 성공 기준이 적용되지 않는 시간 제한의 예로는 반복되는 텍스트 스크롤, 캡션 및 [캐러셀](https://www.w3.org/WAI/tutorials/carousels/)이 있다. 여기에는 시간 제한이 포함되지만 [2.2.2 일시정지, 중지, 숨김](pause-stop-hide)에 지정된 대로 콘텐츠에 대한 접근 제어 기능이 있으므로 사용자는 콘텐츠를 계속 사용할 수 있다.

그러나 시간 제한을 변경할 수 없는 경우(예: 경매 또는 기타 실시간 이벤트)에는 예외가 적용된다.

시간 제한에 의존하지 않는 대안이 있는 경우 타이머에 따라 작동하는 콘텐츠는 시간을 조정할 필요가 없다. 예를 들어, 이메일 클라이언트와 같은 웹 애플리케이션은 인터페이스 오른쪽 하단에 임시 메시지(예: '토스트' 메시지)와 함께 새 이메일이 도착했다는 알림을 제공하며 해당 메시지는 5초 후에 사라진다. 사용자는 받은 편지함 보기 등의 다른 방법을 통해 이메일 도착을 확인할 수 있으므로 메시지가 사라져도 새 메일이 도착했는지 확인하는 데 제한 시간이 없다. 사용자에게 동일한 정보를 검색하거나 동일한 기능을 수행할 수 있는 다른 방법이 없는 경우 사용자가 정보에 접근할 수 있는 충분한 시간을 제공하려면 각 메시지가 이 성공 기준을 충족해야 한다.

**서버 시간 제한에 관한 참고 사항**

*   시간이 지정된 서버 리디렉션은 아래의 일반적인 오류에서 찾을 수 있다.
*   시간 제한이 없는 서버 리디렉션(예: 3xx 응답 코드)은 시간 제한이 없으므로 적용할 수 없다: 즉시 작동한다.
*   이 성공 기준은 콘텐츠 자체에 설정된 시간 제한에만 적용된다. 예를 들어 보안 문제를 해결하기 위해 시간 제한이 포함된 경우 해당 콘텐츠에 대한 프레젠테이션 및 상호 작용 경험의 일부로 설계되었으므로 콘텐츠에 의해 설정된 것으로 간주된다. 사용자 에이전트나 인터넷 고유의 요인 등 콘텐츠 외부에서 설정된 시간 제한은 저작자의 통제를 받지 않으며 WCAG 적합성 요구 사항의 적용을 받지 않는다. 웹 서버에 의해 설정된 시간 제한은 저작자/조직의 통제 하에 있어야 하며 적용된다. (성공 기준 [2.2.3](no-timing), [2.2.4](interruptions) 및 [2.2.5](re-authenticating)도 적용될 수 있다.)
*   임상 경험과 기타 지침을 바탕으로 기본값의 10배를 채택하였다. 예를 들어, 사용자가 응답하고 스위치를 누르는 데 15초가 허용된다면 거의 모든 사용자는 스위치를 누르는 데 문제가 있더라도 150초면 충분하다.
*   20초 역시 임상 경험과 기타 지침을 바탕으로 한 것이다. 경직이 있는 사용자를 포함하여 거의 모든 사용자에게 '아무 스위치'든 누르는 데 20초면 충분하다. 일부는 실패하지만 일부는 긴 시간 동안 실패할 것이다. 일부 애플리케이션의 경우 임의로 오랜 시간이 소요될 경우 장애인을 포함한 모든 사용자에게 보안 위험을 초래할 수 있으므로 추가 시간을 요청하는 합리적인 기간이 필요하다. 예를 들어, 금융 거래에 사용되는 키오스크나 단말기의 경우 사람들이 확인(signing off)을 하지 않고 자리를 떠나는 경우가 매우 흔하다. 이로 인해 그들은 뒤에서 걸어오는 사람들에게 취약해진다. 요청하기 전에 오랜 기간 동안 활동이 없는 상태를 제공하고 그 사람이 현재 있음을 나타낼 때까지 오랜 기간을 제공하면 터미널이 남용될 수 있다. 활동이 없으면 시스템은 사용자가 거기에 있는지 물어봐야 한다. 그런 다음 사람이 거기에 있다는 표시('아무 키나 누르세요')를 요청한 다음 거의 모든 사람이 응답할 때까지 기다려야 한다. "아무 키나 누르세요"의 경우 20초에 해당한다. 사람이 아직 존재한다고 표시하면 장치는 사용자에게 질문을 하기 전의 정확한 상태로 돌아가야 한다.
*   20시간은 깨어 있는 하루보다 길기 때문에 상한선으로 선택되었다.

시간 제한이 본질적인 요구 사항은 아니지만 사용자에게 시간이 지정된 이벤트에 대한 제어 권한을 부여하면 결과가 무효화되는 경우 제3자가 사용자에 대한 제한 시간을 조정할 수 있다(예: 시험에 두 배의 시간 부여).

참조하라 [2.2.3: 시간 제한 없음](no-timing).

### 이점

*   지체 장애가 있는 사람들은 반응하고, 입력하고, 활동을 완료하는 데 더 많은 시간이 필요한 경우가 많다. 시력이 낮은 사람들은 화면에서 사물을 찾고 읽는 데 더 많은 시간이 필요하다. 시각 장애가 있고 화면 낭독기를 사용하는 사람들은 화면 레이아웃을 이해하고 정보를 찾고 컨트롤을 작동하는 데 더 많은 시간이 필요할 수 있다. 인지적 또는 언어적 제한이 있는 사람들은 읽고 이해하는 데 더 많은 시간이 필요하다. 청각 장애가 있고 수어로 의사소통하는 사람들은 텍스트(일부에게는 제2언어일 수 있음)로 인쇄된 정보를 읽는 데 더 많은 시간이 필요할 수 있다.
*   수어 통역사가 청각 장애가 있는 사용자에게 오디오 콘텐츠를 연결하는 상황에서는 제한 시간을 조정하는 ​​것도 중요하다.
*   정보를 읽거나 이해하는 데 더 많은 시간이 필요한 읽기 장애, 인지 장애, 학습 장애가 있는 사용자는 콘텐츠를 일시 중지하여 정보를 읽을 수 있는 추가 시간을 가질 수 있다.

### 예제

*   웹 사이트는 클라이언트측 시간 제한을 사용하여 컴퓨터에서 자리를 비울 수 있는 사용자를 보호한다. 일정 기간 동안 활동이 없으면 웹 페이지는 사용자에게 시간이 더 필요한지 묻는다. 응답을 받지 못하면 시간이 초과된다.
*   웹 페이지에는 순환 방식으로 최신 헤드라인으로 자동 업데이트되는 필드가 있다. 사용자가 각 업데이트 사이의 시간을 기본값의 10배까지 연장할 수 있는 대화형 컨트롤이 있다. 컨트롤은 마우스나 키보드로 작동할 수 있다.
*   웹 페이지에는 전체적으로 나타나고 사라지는 텍스트가 포함된 애니메이션이 포함되어 있다. 어떤 경우에는 텍스트가 화면을 가로질러 스크롤되고 다른 경우에는 배경으로 사라지기 전에 짧은 시간 동안만 표시된다. 페이지에는 텍스트가 사라지기 전에 읽기 어려운 사용자가 읽을 수 있도록 일시 중지 버튼이 포함되어 있다.
*   경매에는 사용자가 입찰을 제출해야 하는 시간에 대한 시간 제한이 있다. 시간 제한은 특정 품목에 입찰하려는 모든 사용자에게 적용되므로 특정 사용자에 대해 시간 제한을 연장하는 것은 불공평하다. 따라서 이러한 유형의 활동에는 시간 제한이 필요하며 이 적합 기준에서는 시간 제한의 연장, 조정 또는 비활성화가 필요하지 않다.
*   온라인 티켓 구매 사이트에서는 좌석이 일반 수영장으로 반환되기 전에 사용자에게 구매를 확인하는 데 2분의 시간을 준다. 이러한 사이트의 티켓은 빨리 매진될 수 있으므로 그 이상 티켓을 보유하면 사이트의 성격이 무효화될 수 있으므로 타이밍이 필수적인 경우이며 활동을 무효화하지 않고는 연장할 수 없다. 그러나 사이트는 시간이 중요한 단계에 들어가기 전에 사용자가 이름, 지불 방법 등과 같은 필요한 정보를 제공할 수 있도록 허용하는 등 가능한 한 많은 작업을 시간이 중요한 기간 외에 처리하도록 이동한다.
*   티켓 구매 사이트에서는 사용자가 선택한 좌석의 구매를 확인하는 데 2분을 허용하지만 시간이 거의 다 되면 사용자에게 경고하고 "시간 연장하기" 버튼을 클릭하는 등의 간단한 동작으로 이 시간 제한을 여러 번 연장할 수 있다.

### 기법

이 섹션의 각 번호가 매겨진 항목은 접근성 가이드라인 실무 그룹이 이 성공 기준을 충족하기에 충분하다고 판단하는 기법 또는 기법의 조합을 나타낸다. 기법은 기준의 최소 요구 사항을 넘어설 수 있다. 이러한 기법에서 포괄하지 않은 기준 충족의 다른 방법이 있을 수 있다. 다른 기법 사용에 대한 정보는 [WCAG 성공 기준에 대한 기법 이해](../understanding-techniques), 특히 "기타 기법" 섹션을 참조하라.

#### 충분 기법

아래에서 귀하의 콘텐츠와 일치하는 상황을 선택하라. 각 상황에는 해당 상황에 충분하다고 알려져 있고 문서화된 기법 또는 기법의 조합이 포함된다.

##### 상황 A: If there are session time limits:

*   [G133: Providing a checkbox on the first page of a multipart form that allows users to ask for longer session time limit or no session time limit](https://www.w3.org/WAI/WCAG22/Techniques/general/G133)
*   [G198: Providing a way for the user to turn the time limit off](https://www.w3.org/WAI/WCAG22/Techniques/general/G198)

##### 상황 B: If a time limit is controlled by a script on the page:

*   [G198: Providing a way for the user to turn the time limit off](https://www.w3.org/WAI/WCAG22/Techniques/general/G198)
*   [G180: Providing the user with a means to set the time limit to 10 times the default time limit](https://www.w3.org/WAI/WCAG22/Techniques/general/G180)
*   [SCR16: Providing a script that warns the user a time limit is about to expire](https://www.w3.org/WAI/WCAG22/Techniques/client-side-script/SCR16) **AND** [SCR1: Allowing the user to extend the default time limit](https://www.w3.org/WAI/WCAG22/Techniques/client-side-script/SCR1)

##### 상황 C: If there are time limits on reading:

*   [G4: Allowing the content to be paused and restarted from where it was paused](https://www.w3.org/WAI/WCAG22/Techniques/general/G4)
*   [G198: Providing a way for the user to turn the time limit off](https://www.w3.org/WAI/WCAG22/Techniques/general/G198)
*   [SCR33: Using script to scroll content, and providing a mechanism to pause it](https://www.w3.org/WAI/WCAG22/Techniques/client-side-script/SCR33)
*   [SCR36: Providing a mechanism to allow users to display moving, scrolling, or auto-updating text in a static window or area](https://www.w3.org/WAI/WCAG22/Techniques/client-side-script/SCR36)

#### 오류

다음은 WCAG 실무 그룹에서 이 성공 기준의 실패로 간주하는 일반적인 실수이다.

*   [F40: Failure due to using meta redirect with a time limit](https://www.w3.org/WAI/WCAG22/Techniques/failures/F40)
*   [F41: Failure of Success Criterion 2.2.1, 2.2.4, and 3.2.5 due to using meta refresh to reload the page](https://www.w3.org/WAI/WCAG22/Techniques/failures/F41)
*   [F58: Failure of Success Criterion 2.2.1 due to using server-side techniques to automatically redirect pages after a time-out](https://www.w3.org/WAI/WCAG22/Techniques/failures/F58)
