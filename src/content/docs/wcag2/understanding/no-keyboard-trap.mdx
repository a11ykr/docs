(Level A)

## 요약

<dl>
<dt>목표</dt>
<dd>키보드 사용자에게 막힘이 없다.</dd>
<dt>할 일</dt>
<dd>사용자가 컴포넌트를 벗어나 탐색하는 방법을 항상 알 수 있도록 하라.</dd>
<dt>중요성</dt>
<dd>키보드에만 의존하는 사람의 경우 다른 탐색수단이 없는 경우가 많다.</dd>
</dl>

<Card title="성공 기준(SC)">
[키보드 인터페이스](../../../glossary/keyboard-interface)를 사용해 키보드 초점을 페이지의 컴포넌트로 이동할 수 있는 경우, 키보드 인터페이스만으로도 해당 컴포넌트에서 초점을 내보낼 수 있어야 한다. 그리고 일반적인 화살표키나 탭 등 표준적인 이동방법 외에 조작이 필요하면 사용자에게 초점 이동법을 안내해야한다.

    <Card title="참고">
    이 성공 기준을 충족하지 못하는 콘텐츠는 전체 페이지를 사용하는 사용자들의 능력을 저해할 수 있다. 웹 페이지의 모든 콘텐츠는 다른 성공기준의 충족 여부와 관계없이 반드시 이 성공기준을 준수해야 한다. [준수 요구사항 5: 불간섭](https://www.w3.org/TR/WCAG22/#cc5)을 참고하라.
    </Card>
</Card>


### 의도
이 성공 기준의 목적은 웹 페이지 콘텐츠의 서브섹션에 키보드 초점이 "갇히는" 일이 없도록 하는 것이다. 여러 형식이 결합된 페이지나, 플러그인이나 임베드된 애플리케이션을 사용하여 렌더링할 때 흔히 발생하는 문제이다.

웹 페이지의 기능상 콘텐츠의 하위 섹션으로 초점을 제한하는 경우도 있을 수 있지만, 사용자가 그 상태를 벗어나 초점을 "빠져 나올 수 있는" 방법을 알고 있다면 허용된다.

### 이점

*   시각 장애인과 지체 장애가 있는 사람을 포함하여 키보드나 키보드 인터페이스를 사용하여 웹을 사용하는 사람들.

### 예제

<dl>
    <dt>달력 위젯</dt>
    <dd>
        달력 위젯을 이용해 사용자가 키보드를 사용하여 달력에 항목을 추가, 제거 또는 업데이트할 수 있다. 위젯의 컨트롤은 웹 페이지 내 탭 순서의 일부이므로 사용자는 위젯의 컨트롤뿐만 아니라 이후에 나오는 모든 링크나 컨트롤을 탭으로 이동할 수 있다.
    </dd>
    <dt>퍼즐 애플릿</dt>
    <dd>
        사용자가 탭으로 애플릿에 들어오면, 추가적인 탭키와 다른 키 입력이 애플릿에 의해서 처리된다. 애플릿을 종료하는 데 사용되는 키 입력 설명이 애플릿 이전과 애플릿 내부에 제공된다.
    </dd>
    <dt>모달 대화 상자</dt>
    <dd>
        웹 애플리케이션이 대화 상자를 표시한다. 대화 상자 하단에는 취소와 확인이라는 두 개의 버튼이 있다. 대화 상자가 열리면, 대화 상자 바깥의 콘텐츠는 비활성화되어 초점을 받지 못한다.(다만, 구현에 따라 초점이 갈 수 있는 범위는 여전히 사용자 에이전트 컨트롤 아래에 있을 수 있다.) 대화 상자는 취소 버튼이나 확인 버튼을 동작시키면 사라진다.
    </dd>
</dl>

### 기법

이 섹션의 각 번호가 매겨진 항목은 접근성 지침 실무 그룹이 이 성공 기준을 충족하기에 충분하다고 판단하는 기법 또는 기법의 조합을 나타낸다. 기법은 기준의 최소 요구 사항을 넘어설 수 있다. 이러한 기법에서 포괄하지 않은 기준 충족의 다른 방법이 있을 수 있다. 다른 기법 사용에 대한 정보는 [WCAG 성공 기준에 대한 기법 이해](../understanding-techniques), 특히 "기타 기법" 섹션을 참조하라.

#### 충분 기법

*   [G21: Ensuring that users are not trapped in content](https://www.w3.org/WAI/WCAG22/Techniques/general/G21)

#### 오류

다음은 접근성 지침 실무 그룹에서 이 성공 기준의 실패로 간주하는 일반적인 실수이다.
