개요
--------

이 문서는 [접근 가능한 리치 인터넷 애플리케이션(WAI-ARIA) 1.2](https://www.w3.org/TR/wai-aria-1.2/) 및 [디지털 퍼블리싱 WAI-ARIA 모듈 1.0](https://www.w3.org/TR/dpub-aria-1.0/) 특성을 \[[HTML](#bib-html "HTML Standard")\] 요소에 사용하는 저작 규칙(저작자 준수 요건)을 정의한다. 이 문서의 주요 목표는 저작자가 웹 콘텐츠를 개발하는 데 사용하는 준수 확인 도구에 필요한 요구 사항을 정의하는 것이다. 이러한 요구 사항은 저작자가 ARIA를 사용하여 호스트 언어 \[[HTML](#bib-html "HTML Standard")\]의 기능을 보완하거나 확장하는 사용자 지정 인터페이스 및 위젯을 개발하는 데 도움이 된다.

1. HTML에서 ARIA 사용에 대한 저작자 요구사항
-----------------------------------------------
[](#rules-wd)

저작자는 [WAI-ARIA](#bib-wai-aria-1.2 "Accessible Rich Internet Applications (WAI-ARIA) 1.2")에서 정의한 요구사항에 따라 [HTML 요소](https://html.spec.whatwg.org/multipage/infrastructure.html#html-elements)의 노출된 의미([의미 체계](https://html.spec.whatwg.org/multipage/dom.html#semantics-2))를 변경하기 위해 ARIA `role`과 `aria-*` 속성을 사용할 수 있다. 단, ARIA 기능이 [강한 원래 의미 체계](https://www.w3.org/TR/wai-aria-1.2/#host_general_conflict)와 충돌하거나 주어진 HTML 요소의 [내재적 ARIA 의미 체계](https://www.w3.org/TR/wai-aria-1.2/#implicit_semantics)와 동일한 경우는 제외한다. HTML 기능의 [내재적 ARIA 의미 체계](https://www.w3.org/TR/wai-aria-1.2/#implicit_semantics)는 [HTML Accessibility API Mappings](#bib-html-aam-1.0 "HTML Accessibility API Mappings 1.0") 명세에서 정의한다.

이 명세에서 정의된 HTML에서 ARIA 기능 사용에 대한 제약사항은 저작자가 보조 기술 제품이 문서의 실제 사용자 인터페이스를 나타내지 않는 비논리적인 사용자 인터페이스 정보를 보고하도록 하는 것을 방지하는 것을 목적으로 한다.

저작자는 [4\. HTML에서 ARIA 속성 사용에 대한 문서 준수 요구사항](#docconformance)과 [4.2 HTML 속성과 동등한 ARIA 속성 사용 요구사항](#docconformance-attr) 표에서 설명된 의미 체계와 충돌하는 방식으로 ARIA `role`과 `aria-*` 속성을 사용해서는 안 된다. 저작자가 두 표에서 정의된 [내재적 ARIA 의미 체계](https://www.w3.org/TR/wai-aria-1.2/#implicit_semantics)와 일치하는 값을 ARIA `role`과 `aria-*` 속성에 설정하는 것은 권장하지 않는다. 이를 수행하는 것은 불필요하며 예기치 않은 결과를 초래할 수 있다.

2. HTML에서 벗어난 ARIA 의미 체계
----------------------------------------------------
[](#aria-semantics-that-extend-and-diverge-from-html)

_이 섹션은 비규범적이다._

ARIA를 사용하면 저작자가 현재 원래 HTML의 기능을 벗어난 의미 체계를 지정할 수 있다. 이는 저작자가 위젯을 만들거나 HTML만으로는 불가능한 원래 HTML 기능에 특정 접근 가능한 상태와 속성을 노출할 수 있는 기회를 제공하기 때문에 매우 유용하다.

예를 들어, `button` 요소는 "눌린" 상태를 노출하는 원래 HTML 기능이 없다. ARIA는 저작자가 `aria-pressed` 속성을 지정하여 요소의 의미 체계를 확장할 수 있게 하여, 시각적 표시와 일치하는 음성 UI를 제공할 수 있다.

다음 예제에서 `button` 요소는 웹 애플리케이션 내 설정의 상태를 전환할 수 있도록 한다. `aria-pressed` 속성이 `button` 요소를 보완한다. "눌린" 상태일 때 해당 정보는 보조 기술 사용자에게 노출될 수 있다.

<Card>
  [예제 1](#example-communicate-a-button-s-pressed-state-with-aria): ARIA를 사용하여 버튼의 눌린 상태를 전달
```html
<button aria-pressed=true>...</button>
```
</Card>
특정 `aria-*` 속성은 다음과 같은 상황에서 허용된다. 특정 `role`을 가진 요소에 사용할 수 있으며, 해당하는 원래 속성이 현재 HTML에서 유효하지 않은 경우에도 사용할 수 있다.

예를 들어, HTML은 비활성화된 하이퍼링크(`a href` 요소)의 개념이 없다. `< a href="..." disabled> ... </a>`와 같은 구조는 유효하지 않으며 보조 기술에 전달되지 않는다.

반면 ARIA는 이 점에서 HTML과 차이가 있으며, 명시적으로 `role=link`를 가진 요소에 `aria-disabled` 속성을 지정할 수 있다. 저작자가 HTML 하이퍼링크에 `aria-disabled=true`를 지정하더라도 사용자는 해당 하이퍼링크를 여전히 클릭하거나 조작할 수 있지만, 보조 기술에는 비활성화된 상태로 전달된다.

마찬가지로, 원래 HTML의 `select`의 자식인 `option` 요소는 `selected`만 설정할 수 있지만, 명시적으로 `option` 역할을 가진 요소는 `aria-selected`와 `aria-checked` 속성을 모두 허용하여, 원래 `select` 요소의 기능을 벗어난 위젯/구조를 지원한다.

불행히도, ARIA와 HTML이 기능적으로 동등하지만 허용 범위가 다른 상황에서는 지원이 일치하지 않거나 사용자 경험도 일관되지 않을 수 있다. ARIA가 HTML에서 지원하지 않는 기능을 허용하는 상황에서는 저작자와 궁극적으로 사용자 모두에게 완전히 사용자 정의된 ARIA 위젯을 구현하는 것이 가장 좋을 수 있다.

다음 예제에서는 하이퍼링크가 비활성화된 상태로 전달되어야 하는 상황을 보여준다. HTML은 하이퍼링크에 `disabled` 속성을 사용할 수 없으며, `aria-disabled=true`를 사용하면 보조 기술에는 비활성화된 상태로 전달되지만 실제로 요소는 비활성화되지 않는다. 하이퍼링크를 비활성화하고 전달하는 가장 효과적인 방법은 [`a`](https://html.spec.whatwg.org/multipage/text-level-semantics.html#the-a-element) 요소에서 `href`를 제거하여 플레이스홀더를 만들고, 그 다음 ARIA를 적용하여 요소의 의도된 역할과 상태를 전달하는 것이다.

<Card>
  [예제 2](#example-communicate-a-disabled-link-with-aria): ARIA를 사용하여 비활성화된 하이퍼링크를 전달
```html
<a role=link aria-disabled=true>...</a>
```
</Card>


3. ARIA의 잘못된 사용을 피하기 위한 저작자 가이드라인
--------------------------------------------------
[](#author-guidance-to-avoid-incorrect-use-of-aria)

_이 섹션은 비규범적이다._

### 3.1 상호작용 가능한 요소의 역할을 비상호작용 가능한 요소의 역할로 대체하지 않기
[](#avoid-overriding-interactive-elements-with-non-interactive-roles)

ARIA는 사용자에게 노출되어야 하는 다른 역할이 필요할 때 요소의 역할을 수정하거나 보정하는 데 유용하다. 그러나 ARIA를 사용하여 예를 들어 `button`과 같은 상호작용 가능한 요소의 역할을 일반적으로 비상호작용 가능한 요소에서 노출되는 역할로 대체하는 것은 사용자나 저작자에게 드문 경우에만 유리하다. 예를 들어, 제목(heading)의 역할로 대체하는 경우이다.

다음은 `button` 요소에 `role=heading`을 사용한 예이다. 이는 허용되지 않는다. `button` 요소는 제목 역할에 대한 사용자 기대와 충돌하는 기본 기능을 가지고 있기 때문이다.

<Card>
  [예제 3](#example-wrong-role): 잘못된 역할
```html
<button role="heading" >search</button>
```
</Card>

저작자는 `button`의 기본 기능과 표현이 제거되도록 추가적인 단계를 취해야 합니다. 그러나 사용자가 웹 페이지와 상호작용하는 방식에 따라 이러한 조치를 취하더라도 요소의 내장 기능을 완전히 제어하기에 충분하지 않을 수 있다. 예를 들어, Windows의 고대비 테마를 켜거나 브라우저의 리더 모드로 웹 페이지를 보는 경우이다.

### 3.2 중복된 역할을 지정하지 않기
[](#avoid-specifying-redundant-roles)

다음 예에서는 [`button`](https://html.spec.whatwg.org/multipage/form-elements.html#the-button-element) 요소에 `role=button`도 명시적으로 추가되어 있다. 이 역할을 지정하는 것은 불필요하며, "button" 요소는 이미 암묵적으로 `button` 역할로 표시되는 것을 의미한다. 실제로는 이러한 중복이 예상치 못한 부작용을 일으키지는 않을 것이지만, 마크업을 불필요하게 만들고, 다른 저작자에게 이러한 예제가 유용하다는 잘못된 신호를 보낼 수 있다. 불필요한 역할을 지정하는 것이 문제가 될 수 있는 예를 확인하려면 [3.3 예기치 못한 부작용에 주의](#side-effects) 섹션을 참조하라.

<Card>
  [예제 4](#example-redundant-role-on-button): 중복된 역할이 지정된 button
```html
<!-- 이렇게 하면 안된다 -->
<button role="button">...</button>
```
</Card>

비슷하게, 다음 예제는 `fieldset` 요소에 `role=group`을, `main` 요소에 `role=Main`을 사용한다. 이는 불필요하다. `fieldset` 요소는 이미 `group` 역할을 내장하고 있으며, `main` 요소도 이미 `main` 역할을 내장하고 있기 때문이다. 마찬가지로, `role` 값의 선언이 [ASCII 소문자](https://infra.spec.whatwg.org/#ascii-lowercase)를 사용하는 한, 실제로 보조 기술을 사용하는 사용자에게는 예상치 못한 부작용이 발생하지 않을 것이다. 자세한 내용은 [4.4 ARIA 역할, 상태, 속성의 대소문자 요구사항](#case-sensitivity)을 참조하라.

<Card>
  [예제 5](#example-redundant-role-on-fieldset-and-main): 중복된 역할이 지정된 fieldset과 main
```html
<!-- 이렇게 하면 안된다 -->
<fieldset role="group">...</fieldset>
<!-- 이렇게 해서도 안된다 -->
<main role="Main" >...</main>
```
</Card>

다음은 `[ul](https://html.spec.whatwg.org/multipage/grouping-content.html#the-ul-element)` 요소에 `role=list`를 사용하는 예시이다. `ul` 요소는 암묵적으로 `list` 역할을 가지고 있으므로, 이를 명시적으로 추가하는 것은 일반적으로 중복으로 간주한다. 그러나 일부 사용자 에이전트에서는 목록 표시가 시각적으로 표시되지 않으면 목록의 암묵적 ARIA 의미를 삭제할 수 있다. 일반적으로 요소의 암묵적 역할을 중복 선언하는 것은 권장되지 않지만 이러한 경우에는, 그리고 역할이 노출되기 위해 필요한 경우에는 저작자가 암묵적 역할을 명시적으로 추가할 수 있다.

<Card>
  [예제 6](#example-redundant-role-on-list): 역할이 중복 지정된 목록
```html
<!-- 일반적으로 이렇게 하면 안된다 -->
<ul role="list" >
	...
</ul>
```
</Card>

### 3.3 부작용에 주의하라
[](#side-effects)

다음 예제는 `summary` 요소에 `role=button`을 사용한다. 이는 불필요하며 플랫폼 간 문제를 일으킬 수 있다. 예를 들어, 요소의 상태를 올바르게 노출하는 것을 방해하고, 플랫폼이나 브라우저 고유의 역할로 노출될 수 있는 상황에서 `button` 역할을 강제할 수 있다.

<Card>
  [예제 7](#example-unintended-consequences): 의도하지 않은 결과
```html
<details>
<!-- 이렇게 하면 안된다 -->
<summary role="button">추가 정보</summary>
	...
</details>
```
</Card>

### 3.4 ARIA의 규칙을 따르라
[](#adhere-to-the-rules-of-aria)

[Accessible Rich Internet Applications (WAI-ARIA) 1.2](https://www.w3.org/TR/wai-aria-1.2/)는 저작자가 사용하도록 설계되지 않은 여러 역할을 정의한다. 이 중 많은 역할들이 저작자가 사용해서는 안 된다고 명시된 [추상적 역할(Abstract Roles)](https://www.w3.org/TR/wai-aria-1.2/#isAbstract)로 분류한다. 다음 예제는 저작자가 `combobox` 역할을 사용하려고 했을 가능성이 있는 추상적 `select` 역할의 잘못된 사용을 보여준다.

<Card>
  [예제 8](#example-abstract-roles-are-not-for-authors): 추상적 역할은 저작자용이 아닙니다
```html
<!-- 이렇게 하면 안된다 -->
<div role="select" ...>...</div>
```
</Card>

ARIA는 또한 HTML의 `div`와 `span` 요소와 같은 특정 ARIA 의미론을 가지고 있지 않은 여러 요소와 기능을 동일하게 만드는 [`generic` 역할](https://www.w3.org/TR/wai-aria-1.2/#generic)을 정의헌다. ARIA는 `generic` 역할을 사용자 에이전트 구현자용으로 설계되었기 때문에 저작자가 사용하는 것을 권장하지 않는다.

다음 예제는 `article` 요소에 `generic` 역할을 사용하는 대신 `div` 요소를 사용하는 것을 저작자에게 권장한다. HTML 요소를 변경할 수 없는 경우 `presentation` 또는 `none` 역할을 지정하여 `article`의 내장 역할을 제거하는 것이 허용된다.

<Card>
  [예제 9](#example-do-not-specify-elements-as-generic): 요소를 generic으로 지정하지 말라
```html
<!-- 이렇게 하면 안된다 -->
<article role="generic" ...>...</article>
```
</Card>

이외에, ARIA는 [주 언어 의미와 충돌](https://www.w3.org/TR/wai-aria-1.2/#host_general_conflict)에서 저작자가 상태 및 속성을 노출하는 HTML의 기본 기능과 ARIA의 대응하는 기능을 함께 사용할 경우, 명시적으로 ARIA 속성을 사용할지라도 주 언어의 기능이 우선순위를 가짐을 언급한다.

예를 들어, 다음 예제에서는 저작자가 HTML의 `input type=checkbox`를 사용하고 `aria-checked=true`를 지정하였다. 하지만, 사용자 에이전트는 `aria-checked` 속성을 무시해야 하며, 대신 폼 컨트롤의 기본 기능을 기반으로 상태를 노출해야 한다.

<Card>
  [예제 10](#example-the-implicit-checked-state-takes-precedent-over-the-explicit-aria-attribute): 암묵적 체크 상태가 명시적 ARIA 속성보다 우선한다
```html
<!-- 이렇게 하면 안된다 -->
<input type="checkbox" checked aria-checked="false">
```
</Card>

### 3.5 HTML의 규칙을 따르라
[](#adhere-to-the-rules-of-html)

ARIA는 보조 기술 사용자에게 HTML 기능을 노출하는 방식을 변경하는 데 사용될 수 있지만, 이러한 수정은 HTML의 기본 파싱과 허용된 콘텐츠 모델을 변경하지 않는다. 예를 들어, `div` 요소는 저작자가 어떤 역할도 지정할 수 있다. 그러나 이는 요소가 HTML이 정의한 규칙을 위반하는 방식으로 사용될 수 있다는 의미는 아니다.

예를 들어, 다음 예제에서는 저작자가 `div` 요소에 `link` 역할을 지정했다. HTML은 하이퍼링크(`role=link`로 노출)가 `p` 요소의 자식이 될 수 있지만, HTML 파서는 `p` 요소의 자식으로 `div` 요소를 허용하지 않는다.

<Card>
  [예제 11](#example-revised-aria-semantics-with-invalid-html-nesting): ARIA 의미론을 수정한 후 HTML 상하 구조(nesting)가 잘못된 경우
```html
<!-- 이렇게 하면 안된다 -->
<p>
	... <div role=link tabindex=0>...</div> ...
</p>
```
</Card>

HTML 파서는 위의 마크업을 다음과 같이 수정하여 출력한다.

<Card>
  [예제 12](#example-unwanted-rendered-markup-with-valid-alternative-solution): 원치 않게 렌더링된 마크업과 유효한 대체 솔루션
```html
<!-- 이전 예제의 마크업은 다음과 같이 렌더링된다 -->
<p>
	...
</p>
<div role=link tabindex=0>...</div>
...
<p></p>

<!-- div 대신 span 요소를 사용하라 span 요소는 p 요소의 자손으로 허용된다. -->
<p>
	...<span role=link tabindex=0>...</span>...
</p>
```
</Card>

이러한 예에서는 ARIA 속성이 HTML 요소에 지정될 수 있는지 여부를 나타내지만, 이러한 속성이 사용되는 문맥 자체가 여전히 잠재적인 표시 및 접근성 문제를 초래할 수 있음을 보여준다.

4\. HTML 문서에서 ARIA 속성 사용에 대한 문서 준수 요구사항
------------------------------------------------------------------------
[](#docconformance)

다음 표는 HTML 문서에서 ARIA 마크업 사용에 대한 문서 준수 요구사항을 정의하고, HTML 요소에 대한 암묵적 ARIA 의미론을 식별한다. 또한, 이러한 요소에 대한 암묵적 ARIA 의미론은 [HTML AAM](#bib-html-aam-1.0 "HTML Accessibility API Mappings 1.0")에서 정의한다.

  각 언어 기능(요소)은 첫 번째 열의 셀에서 두 번째 열의 셀에 주어진 ARIA 의미론(역할, 상태 및 속성)을 암시합니다. 각 행의 세 번째 셀은 저작자가 요소에 지정할 수 있는 ARIA `role` 값과 `aria-*` 속성을 정의합니다. 세 번째 열의 셀에 **Any**`role`이라는 용어가 포함된 경우, 저작자가 요소에 사용할 수 있는 모든 `role` 값이 _MAY_ 지정될 수 있음을 나타냅니다. 그러나 저작자가 요소의 내재적 역할, `generic` 역할, 또는 ARIA에 의해 [비권장](#docconformance-deprecated)된 역할을 지정하는 것은 _권장하지 않음_입니다. 세 번째 열의 셀에 **No `role`**이라는 용어가 포함된 경우, 저작자가 내재적 ARIA 의미론이나 HTML 요소의 고유 의미론을 덮어쓰는 것은 _MUST NOT_입니다.

<div class="addition">
	[WAI-ARIA](#bib-wai-aria-1.2 "Accessible Rich Internet Applications (WAI-ARIA) 1.2")는 저작자가 지정할 수 없는 [금지된 상태 및 속성](https://www.w3.org/TR/wai-aria-1.2/#prohibitedattributes)을 가진 역할을 식별합니다. 이러한 역할은 저작자가 특정 WAI-ARIA 속성을 지정하는 것을 허용하지 않습니다. 이러한 내재적 WAI-ARIA 역할을 노출하는 HTML 요소도 저작자가 이러한 WAI-ARIA 속성을 지정하는 것을 허용하지 않습니다.

	저작자가 명명할 수 없는 요소는 `aria-label`혹은 `aria-labelledby`속성을 지정할 수 없는 요소이다. 이러한 요소는 암묵적 역할을 저작자가 명시적인 WAI-ARIA 역할로 덮어쓰지 않는 한, 명명할 수 없다. 자세한 정보는 [4.1 ARIA 속성을 사용하여 요소 명명](#docconformance-naming)참고.
</div>

<Aside title="참고">
	암묵적 ARIA 의미와 일치하는 ARIA `role`및 `aria-*`속성을 설정하는 것은 _권장하지 않는다_. 그러나 일부 상황에서는 이러한 속성을 명시적으로 설정하는 것이 도움이 될 수 있다. 예를 들어, 특정 요소에 대한 암묵적 ARIA 의미론을 노출하지 않는 사용자 에이전트의 경우가 있다.
</Aside>

<Aside title="참고">
다음 표에 정의된 [Digital Publishing(Dpub) WAI-ARIA Module 1.0](https://www.w3.org/TR/dpub-aria-1.0/) `role` 값을 사용하는 것은 적합하지만, 이러한 역할을 웹 사이트 구현 목적으로 사용하는 것은 의도와 맞지 않는다. 디지털 출판 산업 이외의 목적으로 이러한 역할을 사용하는 경우에는 사용자에게 의도된 경험을 제공하는지 확인하기 위해 추가적인 테스트가 필요하다.

### 4.2 Requirements for use of ARIA attributes in place of equivalent HTML attributes

[](#docconformance-attr)

Unless otherwise stated, authors _MAY_use `aria-*`attributes in place of their HTML equivalents on HTML elements where the `aria-*`semantics would be expected. For example, authors _MAY_specify `aria-disabled=true`on a [`button`](https://html.spec.whatwg.org/multipage/form-elements.html#the-button-element)element, while also implementing the necessary scripting to functionally disable the `button`, rather than the use `disabled`attribute.

As stated in [WAI-ARIA's Conflicts with Host Language Semantics](https://www.w3.org/TR/wai-aria-1.2/#host_general_conflict), when HTML elements use _both_`aria-*`attributes and their host language (HTML) equivalents, user agents _MUST_ignore the WAI-ARIA attributes – the native HTML attributes with the same [implicit ARIA semantics](https://www.w3.org/TR/wai-aria-1.2/#implicit_semantics)take precedence. For this reason, authors _SHOULD NOT_specify both the native HTML attribute and the equivalent `aria-*`attribute on an element. Please review each attribute for any further author specific requirements.

### 4.3 Requirements for deprecated ARIA role, state and property and attributes

[](#docconformance-deprecated)

The ARIA Specification's [Deprecated Requirements](https://www.w3.org/TR/wai-aria-1.2/#deprecated) section indicates that if an ARIA feature is marked as deprecated then authors are advised not to use said feature for new content.

The following roles and attributes are deprecated features of ARIA and DPub ARIA. Conformance checkers _MUST_warn authors about the deprecated status of these features. Whenever possible, authors are advised to use alternatives to deprecated features.

#### 4.3.1 Deprecated ARIA roles

[](#deprecated-aria-roles)

*   [`directory`](https://www.w3.org/TR/wai-aria-1.2/#directory)

Note

The `directory`role is marked for deprecation in [WAI-ARIA 1.2](#bib-wai-aria-1.2 "Accessible Rich Internet Applications (WAI-ARIA) 1.2"). In reality, the `directory`role had no functional difference to an element with an implicit 또는explicit `list`role. Authors are advised to use one of HTML's native list elements, 또는an ARIA `list` instead.

#### 4.3.2 Deprecated DPub ARIA roles

[](#deprecated-dpub-aria-roles)

*   [`doc-biblioentry`](https://www.w3.org/TR/dpub-aria-1.1/#doc-biblioentry)
*   [`doc-endnote`](https://www.w3.org/TR/dpub-aria-1.1/#doc-endnote)

Note

The `doc-biblioentry`and `doc-endnote`roles are marked for deprecation in [Digital Publishing WAI-ARIA Module 1.1](https://www.w3.org/TR/dpub-aria-1.1/), as they are not valid children for an element with an implicit 또는explicit role of `list`. Authors can use standard list and child `li`elements without the need for these roles.

#### 4.3.3 Deprecated ARIA attributes

[](#deprecated-aria-attributes)

*   [`aria-dropeffect`](https://www.w3.org/TR/wai-aria-1.1/#aria-dropeffect)
*   [`aria-grabbed`](https://www.w3.org/TR/wai-aria-1.1/#aria-grabbed)

### 4.4 Case requirements for ARIA role, state and property attributes

[](#case-sensitivity)

Authors _SHOULD_use [ASCII lowercase](https://infra.spec.whatwg.org/#ascii-lowercase)for all `role`token values and any state 또는property attributes (`aria-*`) whose values are [defined as tokens](https://www.w3.org/TR/wai-aria-1.2/#propcharacteristic_value).

Note

While modern browsers treat the `role`or `aria-*`attribute values as [ASCII case-insensitive](https://infra.spec.whatwg.org/#ascii-case-insensitive), not all assistive technologies will correctly parse these values.

To reduce interoperability issues, authors are strongly encouraged to use [ASCII lowercase](https://infra.spec.whatwg.org/#ascii-lowercase)for `aria-*`and `role`attribute values. Further, authors are encouraged to rigorously test with different browser and assistive technology combinations to ensure that their content will be correctly exposed to their users.

5\. Allowed descendants of ARIA roles
-------------------------------------

[](#allowed-descendants-of-aria-roles)

_This section is non-normative._

The following table maps (and extends) the [Kinds of content](https://html.spec.whatwg.org/multipage/dom.html#kinds-of-content)and allowed descendant information (defined in the \[ [HTML](#bib-html "HTML Standard") \] specification) to elements that have an equivalent `role`.

Column 1 links to the normative [Accessible Rich Internet Applications (WAI-ARIA) 1.2](https://www.w3.org/TR/wai-aria-1.2/)definitions for each ARIA `role`. Column 2 identifies the [Kinds of content](https://html.spec.whatwg.org/multipage/dom.html#kinds-of-content)categories each `role`has when it is used on an HTML element. Column 3 indicates what kinds of HTML elements can be descendants of an element with an explicit `role`specified, often matching the HTML element with the same [implicit](#implicit)role.

For example, a [`button`](https://html.spec.whatwg.org/multipage/form-elements.html#the-button-element)element has an implicit `role=button`. In HTML a `button`element allows [phrasing content](https://html.spec.whatwg.org/multipage/dom.html#phrasing-content-2)as descendants, and does not allow [interactive content](https://html.spec.whatwg.org/multipage/dom.html#interactive-content-2)or descendants with a `tabindex`attribute. Therefore, any elements specified with a `role=button`would follow the same descendant restrictions, and not allow any interactive content descendants, elements with a `tabindex`specified, 또는any elements with role values that are in the interactive content category (identified in column 3).

<Card title="예제">
**Examples of non-conforming descendants**

```html
<!-- conformance checkers will report an error -->
<button>
	<div role="button" >...</div>
</button>
<div role="button" >
	<button>...</button>
</div>
<div role="link" >
	<textarea>...</textarea>
</div>
```

6\. Conformance
---------------

[](#conformance)

As well as sections marked as non-normative, all authoring guidelines, diagrams, examples, and notes in this specification are non-normative. Everything else in this specification is normative.

The key words _MAY_, _MUST_, _MUST NOT_, _권장하지 않음_, _SHOULD_, and _SHOULD NOT_in this document are to be interpreted as described in [BCP 14](https://www.rfc-editor.org/info/bcp14)\[[RFC2119](#bib-rfc2119 "Key words for use in RFCs to Indicate Requirement Levels")\] \[[RFC8174](#bib-rfc8174 "Ambiguity of Uppercase vs Lowercase in RFC 2119 Key Words")\] when, and only when, they appear in all capitals, as shown here.

### 6.1 Conformance checking requirements

[](#conformance-checking-requirements)

Conformance checkers that claim support for checking ARIA in HTML documents _MUST_implement checks for the conformance requirements for use of the ARIA `role`and `aria-*`attributes on [HTML elements](https://html.spec.whatwg.org/multipage/infrastructure.html#html-elements)as defined in this specification.

A conforming document _MUST NOT_contain any elements with author defined `role`or `aria-*`attributes with values other than those which, per this specification, authors _MAY_use on each [HTML element](https://html.spec.whatwg.org/multipage/infrastructure.html#html-elements)in [4\. Document conformance requirements for use of ARIA attributes in HTML](#docconformance). Conformance checkers _SHOULD_flag instances where authors are explicitly providing an element with a `role`which matches its [implicit ARIA semantics](https://www.w3.org/TR/wai-aria-1.2/#implicit_semantics)as failures, as it is _권장하지 않음_for authors to explicitly set these roles.

A conformance checker _MAY_define their own terminology, and level 또는levels of severity, when surfacing document failures to conform to this specification.

7\. Privacy and security considerations
---------------------------------------

[](#priv-sec)

_This section is non-normative._

This specification does not define the features of \[[wai-aria-1.2](#bib-wai-aria-1.2 "Accessible Rich Internet Applications (WAI-ARIA) 1.2")\], \[[dpub-aria-1.0](#bib-dpub-aria-1.0 "Digital Publishing WAI-ARIA Module 1.0")\] 또는\[[HTML](#bib-html "HTML Standard")\]. Rather it provides rules and guidance for conformance checkers thatclaim support for checking ARIA in HTML, as well as providing guidance toauthors.

Therefore, there are no known privacy 또는security impacts of this specification, as it defines no new features to introduce potential concern.
